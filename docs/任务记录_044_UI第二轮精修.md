# 任务记录 044：HireMate Streamlit UI 第二轮精修

## 01 任务目标

- 在不推翻现有 Streamlit 主链路的前提下，把 `app.py` 里的“岗位配置页”和“批量初筛页”做成更接近 Stitch 设计稿的内部布局。
- 保持现有产品原则和业务边界不变：
  - 规则评分器为主
  - AI reviewer 为建议层
  - 人工最终决策不被替代
  - 不改业务库 schema

## 02 当前进展

- 已在 `app.py` 中新增更后置的一层 `_render_job_library` / `_render_batch_screening` 覆盖函数，使最终生效版本改为新的深层布局。
- 已新增页面级 UI 组件样式：
  - `ui-surface`
  - `job-tile`
  - `ui-mini-stat`
  - `ui-rail-item`
  - `ui-divider`
- 岗位配置页已改成：
  - 左侧岗位卡片列表
  - 右侧详情配置面板
  - 详情区内使用 `Core Setup / AI Defaults / Batch History` 分区
  - 页面底部保留 `Create New Requisition`
- 批量初筛页已改成：
  - 主操作区：岗位上下文 -> reviewer 配置 -> OCR / 上传 -> 提取预检查 -> 开始批量初筛
  - 右侧状态轨道：当前批次状态 / 最近批次 / 操作提示
- `python -m compileall app.py src` 已通过。

## 03 已做决策 + 理由

- 决策：继续使用“后置覆盖函数”策略，而不是直接重写最早的原始大函数。
  - 理由：当前 `app.py` 业务逻辑很重，继续沿用覆盖层更安全，也更符合上一轮 UI 接入方式。
- 决策：岗位配置页右侧改为分区式 detail panel，而不是继续单列长滚动。
  - 理由：更接近设计稿，也能减少 HR 在一个超长页面里来回滚动。
- 决策：批量初筛页继续保留“pending key + widget 前置 hydrate”的时序。
  - 理由：避免回归到之前 `batch_jd_text_area` 与 `batch_ai_reviewer_*` 的 Streamlit 状态异常。
- 决策：右侧 rail 只放状态、历史和操作提示，不放核心配置控件。
  - 理由：让主操作区更聚焦，符合工作台式布局。

## 04 待解决问题

- 不让 AI reviewer 直接替代人工最终决策
