# HireMate 短期升级方案（以 RAG 降低幻觉为主线）

本方案以“规则评分器为主、AI reviewer 为建议层”为产品原则，RAG 仅作为检索增强和证据 grounding，不直接替代最终结论。

## 01 目标与范围

- 目标
  - 降低 AI reviewer 幻觉与不精确回答
  - 提升 JD 对齐、证据抽取、解释一致性
  - 保持评分主链路稳定
- 范围
  - 不改数据库 schema
  - 不替换规则评分器
  - RAG 不直接产出最终通过/待复核/淘汰结论

## 02 当前基础（已具备）

- 已有 RAG 目录骨架与检索接口（`src/rag/*`）
- 已有本地向量库存储目录约定（`/app/data/vector_store`）
- 已有 RAG 运维手册（[RAG向量库操作手册.md](/D:/apps/HireMate/docs/RAG向量库操作手册.md)）

## 03 短期升级主线（分三段）

### 阶段 A：正式 benchmark + 历史语料索引扩充

- 建立可复现的 benchmark 集
  - 真实 JD + 历史简历 + 证据片段集合
  - 统一评估指标：召回命中率、证据可读性、解释一致性
- 扩充历史语料索引
  - 先以“历史简历库 + 典型 JD + 标注证据”做初始基线
  - 形成“无 RAG / 有 RAG”对比报告

输出物：
- benchmark 数据说明与统计
- 基线对比报告（召回 / 解释一致性）

### 阶段 B：批次与候选人增量索引

- 批次级 / 候选人级增量索引
  - 在当前批次内补充“候选人局部证据库”
  - 让 evidence grounding 不只依赖历史库
- 建立“索引更新触发点”
  - 新批次创建 / 新简历导入 / 关键证据更新

输出物：
- 增量索引策略说明
- 批次级检索调试日志

### 阶段 C：Rerank 轻量调参与可解释提升

- 对检索结果进行轻量重排
  - 语义相似 + 关键词命中 + 技能命中 + source_type 权重
- 证据统一呈现
  - 让“关键证据片段 / 五维证据 / 淘汰理由”口径一致

输出物：
- rerank 权重建议与默认值
- evidence 统一展示样式与差异说明

## 04 具体短期升级建议（可落地任务）

1. 建立“正式 benchmark 目录 + 数据说明”
2. 历史语料索引构建脚本与执行文档
3. 加入“批次增量索引”与基本触发规则
4. 增加检索评估输出（命中率 / 可读性 / 去重率）
5. rerank 首轮参数 sweep（小范围）

## 05 里程碑与验收

- M1：benchmark 可跑通，有基线数据
- M2：历史语料索引可复用
- M3：批次增量索引可用
- M4：rerank 调参有收益

## 06 风险与边界

- 不引入“RAG 直接改写评分结果”的路径
- 不改变五维结构与现有业务主链路
- 只在证据、检索与解释层做增强
