# 任务记录 048｜API 合同收口与前端适配层

## 01 任务目标

- 在不改业务数据库 schema 的前提下，把前后端重构从“可运行骨架”推进到“接口合同稳定、页面消费结构清晰”的阶段。
- 为岗位详情页、批量初筛页、候选人工作台、管理员页建立稳定的 response DTO / view model。
- 让前端页面不再直接依赖 legacy 中文字段和深层原始 payload。
- 保持规则评分器为主、AI reviewer 为建议层、人工最终决策不被替代。

## 03 已做决策 + 理由

- 继续沿用 FastAPI + React/Vite/Tailwind 架构，不回退到继续深修 Streamlit UI。
  - 理由：独立前后端方向已经成型，现在最值钱的是收口合同和联调，而不是重新选型。
- 保留 `src/*` 作为领域与数据层，不迁移业务主逻辑。
  - 理由：可以最大化复用现有解析、评分、筛选、AI reviewer、RAG、存储能力，降低回归风险。
- `job detail` 中保留 `scoring_config` 原始 JSON。
  - 理由：高级配置编辑仍需要兼容现有结构；同时新增结构化字段供页面主视图消费，避免页面直接依赖 raw config。
- `workbench` 路由新增稳定枚举码到 legacy 中文标签的翻译层。
  - 理由：前端可以统一用 `pending_review / priority_desc / high` 这类稳定值，后端继续兼容现有候选池与筛选逻辑。
- 增加“最后一个活跃管理员保护”。
  - 理由：这是管理员页最小但高价值的安全边界，且改动小、回归风险低。

## 04 待解决问题

- Docker / Caddy / HTTPS 同域登录链路还没有在真实 Docker 环境完成联调验活。
- 前端目前仍有少量 snake_case API 字段直接出现在 request/adapter 层，这一层后续可以继续抽象。
- `job detail` 的高级 JSON 编辑仍然偏工程化，后续可继续拆成更友好的权重/AI 默认配置编辑器。
- 需要继续补部署验证：
  - `docker compose config`
  - `docker compose build hiremate-api hiremate-web`
  - `docker compose up -d hiremate-api hiremate-web caddy`

## 05 任务特有约束

- 不改 MySQL / SQLite 业务 schema。
- 不替换规则评分器。
- 不让 AI reviewer / RAG 直接替代人工最终结论。
- 不破坏现有批次级 AI reviewer 配置与人工审核留痕。
- 允许保留少量兼容层，以适配现有 legacy 中文字段和存量数据。
