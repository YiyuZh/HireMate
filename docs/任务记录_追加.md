## 任务记录 041：网关层 Basic Auth 轻量防爆破

### 01 任务目标

- 按用户要求，把后台 Basic Auth 密码箱/暴力破解防护放在 `HireMate` 所属公网网关层或服务器 `fail2ban`，不放进 `portal` 静态页面。
- 保持轻量：不新增重型安全服务，不引入数据库，不改后台账号密码机制。

### 02 当前进展

- 已在 `Caddyfile` 新增 `admin_auth_access_log` snippet，将相关站点访问日志写入 `/var/log/caddy/access.log`。
- 已在 `{$PRIMARY_DOMAIN}` 与 `blog.zenithy.art` 站点块引入访问日志，覆盖：
  - `/api/admin/messages*`
  - `/admin`
  - `/admin/*`
- 已在 `docker-compose.yml` 为 `hiremate-caddy` 增加宿主机日志挂载：`./logs/caddy:/var/log/caddy`。
- 已在 `.gitignore` 忽略 `logs/`，避免访问日志进入 Git。
- 已新增 `security/fail2ban/`：
  - `filter.d/zenithy-caddy-admin-auth.conf`
  - `jail.d/zenithy-caddy-admin-auth.local`
  - `install.sh`
  - `README.md`

### 03 已做决策 + 理由

- 决策：用 Caddy JSON access log + fail2ban 识别 `401`。
  - 理由：Caddy Basic Auth 拒绝请求发生在网关层，portal 前端拿不到失败密码尝试，fail2ban 是最小可行方案。
- 决策：只匹配后台和 admin messages API 的 `401`，不按普通 404/5xx 封禁。
  - 理由：避免误封普通访客，策略更克制。
- 决策：默认 `10m` 内失败 `6` 次封禁 `1h`。
  - 理由：允许用户偶尔输错密码，同时挡住脚本/密码箱高频尝试。

### 04 待解决问题

- 需要在服务器安装并启用 `fail2ban`：
  - `sudo apt install -y fail2ban`
  - `sudo sh security/fail2ban/install.sh`
- 需要服务器执行 `docker compose up -d --force-recreate caddy` 后确认 Caddy 能写入 `/opt/apps/hiremate/logs/caddy/access.log`。
- 如果 Caddy 容器无法写日志，优先检查 `logs/caddy` 目录权限，可执行 `sudo chown -R 1000:1000 logs/caddy || true`。

### 05 任务特有约束

- 不在 `portal` 静态页里做 Basic Auth 爆破防护。
- 不改变后台 Basic Auth 账号密码。
- 不改变公网入口归属，Caddy 事实来源仍为 `D:/apps/HireMate/Caddyfile`。
