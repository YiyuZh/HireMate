最后更新：2026-03-24

## 0. 项目协作约定

本文件作为 HireMate 项目的任务记录基线文档使用。

- 约定 1：后续在本仓库内继续修改时，优先在本文件追加“任务记录”。
- 约定 2：每次记录尽量包含：任务目标、当前进展、已做决策与理由、待解决问题、任务特有约束、出现的错误、修改点、验证方式。
- 约定 3：涉及安全信息时不记录明文密码、明文 API Key、数据库敏感内容。
- 约定 4：涉及 AI 审核时继续遵守“规则评分器为主，AI reviewer 为建议层，不直接替代人工最终结论”。

说明：

- 该约定已写入项目文档，后续协作时应优先遵守。
- 它是项目级约定，不等同于系统级永久指令；但在本仓库继续协作时，默认按此执行。

## 技术栈总览

- 应用层：Python 3、Streamlit
- 核心业务：岗位配置、批量初筛、候选人工作台、规则评分器、AI reviewer 建议层
- 数据层：SQLite / MySQL 双后端，数据库入口集中在 `src/db.py`
- MySQL 驱动：PyMySQL + `cryptography`
- Web 反向代理：Caddy
- OCR 栈：Pillow、pytesseract、pdf2image、tesseract、poppler
- AI 接入：`openai`、`openai_compatible`、`deepseek`、`azure_openai`、`anthropic`、`mock`
- 质量工具：`scripts/smoke_app_flow.py`、`scripts/ocr_regression_check.py`
- 部署方式：Docker / Docker Compose
- 规划中的检索增强层：向量库 RAG（建议 MVP 为本地持久化向量库 + Embedding 模型 + 结构化 metadata 过滤）

## HireMate 下一阶段升级方案（RAG 建议版）

### 阶段 0：先补评估基线

- 先建立一组固定 benchmark：按岗位模板收集一批脱敏简历样本、标准 JD、期望分数区间、关键证据点、人工结论。
- 先量化当前系统的基线表现，后续才知道 RAG 是否真的提高了“标准化”和“一致性”。

### 阶段 1：沉淀可检索语料

- JD 侧：原始 JD、解析后的 must-have / bonus / role profile、评分细则说明。
- 简历侧：清洗稿、结构化片段、关键证据片段、维度代表证据。
- 规则侧：角色模板关键词、维度评分规则、淘汰/复核说明模板。
- 历史侧：脱敏后的优质人工结论与代表性证据样本。

### 阶段 2：做结构化切块与 metadata

- 建议切块单位不要只按固定字数切，而是优先按“教育 / 实习 / 项目 / 技能 / 总结”结构切块。
- 每个 chunk 至少带这些 metadata：
- `source_type`：jd / role_profile / resume_fragment / evidence / rubric
- `role_family`：AI PM / PM / 数据分析 / 用户研究等
- `skill_tags`
- `seniority`
- `language`
- `candidate_id` 或 `job_id` 的安全映射标识

### 阶段 3：接入向量检索层

- MVP 建议采用本地持久化向量库，路径可独立于业务库，例如 `/app/data/vector_store`。
- 检索时先做 metadata 过滤，再做相似度召回，避免“跨岗位乱召回”。
- 召回对象建议分两路：
- 规则与 rubric 路：给评分解释和模板校准用
- 简历证据路：给 evidence 抽取和 AI reviewer grounding 用

### 阶段 4：先接辅助层，不直接接最终结论

- 可优先接入这些位置：
- JD 解析后，为 required / bonus skills 做同义词扩展与标准化
- 经历维度评分前，为岗位模板召回相应“高质量方法/产出/结果”样例词典
- 关键证据片段抽取时，用召回结果帮助定位更相关片段
- AI reviewer 提示词中注入岗位模板解释、相似样例证据和评分依据
- 暂不建议第一阶段就让 RAG 直接改写五维分数或最终筛选结果

### 阶段 5：回归与灰度

- 增加一组离线回归脚本，比较“无 RAG / 有 RAG”在以下指标上的变化：
- 相关经历误抬分率
- JD 技能命中解释一致性
- 关键证据片段相关性
- `screening_reasons` 的可解释性
- 先按岗位模板灰度，比如先从 AI PM 和数据分析两类岗位试点。

### 推荐技术栈（建议，不代表已落地）

- 应用层：继续沿用 Python / Streamlit，不增加新的前端主框架。
- 向量层：优先选本地持久化向量库作为 MVP，后续再视规模升级到独立服务。
- Embedding 层：优先选可在当前 AI provider 体系外独立控制的 Embedding 能力，避免与 reviewer 调用完全耦合。
- 数据治理：继续依赖当前结构化解析、OCR 清洗稿、证据片段体系，不额外引入大规模 schema 改造。

### 为什么 RAG 值得做

- 当前规则链路已经能给分，但对同义技能、岗位变体、复杂经历表达的适配还不够柔性。
- RAG 更适合做“岗位语义对齐”和“证据召回标准化”，能减少同一能力因表述不同导致的误判。
- 只要边界守住，RAG 不会削弱规则评分器，反而会让评分依据更稳定、解释更统一。
