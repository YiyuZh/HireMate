# 任务记录 050｜构建加速与镜像源确认

## 01 任务目标
- 排查 `docker compose up -d --build` 构建过慢是否因为没有使用国内镜像源。
- 在确认镜像源配置无误的前提下，进一步优化 HireMate 的 Docker 构建速度。

## 02 当前进展
- 已确认 `pip` 实际走的是清华 PyPI：
  - `PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple`
  - 构建日志中也明确出现了 `https://pypi.tuna.tsinghua.edu.cn/...`
- 已确认 apt 也配置为清华 Debian 镜像：
  - `APT_MIRROR_HOST=mirrors.tuna.tsinghua.edu.cn`
- 已完成构建链路优化：
  - `hiremate-legacy` 与 `hiremate-api` 改为共用同一个根目录 `Dockerfile`
  - 使用 `target: legacy` / `target: api` 共享 Python 依赖基础层
  - 开启 BuildKit cache mount，加速 apt 与 pip 的重复构建

## 03 已做决策 + 理由
- 保留清华镜像源配置，不切回官方源。
  - 理由：当前下载日志已经证明清华源生效，问题不在“没走国内源”。
- 把 `hiremate-api` 的构建切到根 `Dockerfile` 多阶段目标，而不是继续单独走 `backend/Dockerfile`。
  - 理由：之前 `hiremate-legacy` 和 `hiremate-api` 都各自安装一整套 Python 依赖，重复最重的一层恰好是 `streamlit / pandas / pyarrow`，这是构建慢的主要来源。
- 同时保留 `backend/Dockerfile` 的独立可用性，并补上相同的 pip/apt cache mount。
  - 理由：避免手工单独构建 API 镜像时重新退回慢路径。

## 04 本轮修改文件
- `Dockerfile`
- `backend/Dockerfile`
- `docker-compose.yml`

## 05 修改内容摘要
- 根 `Dockerfile`
  - 改为多阶段：
    - `python-base`
    - `legacy`
    - `api`
  - 在 `python-base` 一次性安装：
    - `requirements.txt`
    - `backend/requirements.txt`
  - 启用：
    - `--mount=type=cache,target=/var/cache/apt`
    - `--mount=type=cache,target=/var/lib/apt/lists`
    - `--mount=type=cache,target=/root/.cache/pip`
- `docker-compose.yml`
  - `hiremate-legacy` 改为：
    - `dockerfile: Dockerfile`
    - `target: legacy`
  - `hiremate-api` 改为：
    - `dockerfile: Dockerfile`
    - `target: api`
- `backend/Dockerfile`
  - 继续保留单独构建能力
  - 同步补上 pip/apt cache mount 与清华源环境变量

## 06 问题定位结论
- 本次构建慢，不是因为没用清华源。
- 真正拖慢的核心是：
  1. `hiremate-legacy` 和 `hiremate-api` 之前各自完整安装一遍 Python 大依赖
  2. `streamlit` 依赖链很重，尤其 `pyarrow` 包较大
  3. 之前 `pip install --no-cache-dir` 无法利用 BuildKit pip 缓存
